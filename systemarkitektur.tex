\section{Systemarkitektur}

Systemarkitekturen skal forbinde brugerflade, state, datamodel og backend, så Træningsmester kan håndtere både planlagt og faktisk træning. Figur~\ref{fig:tm-system-context} viser systemet som en native iOS-app med watchOS, Live Activity, Supabase og udvalgte Apple-services. Arkitekturen vurderes på tre valg: lagdeling, backend som sikkerhedsgrænse og støtteflader under træning.

Arkitekturen er derfor ikke kun en teknisk beskrivelse. Den afgør, om appen kan bevare samme træningsforløb på tværs af Home, tracker, database, Apple Watch og Live Activity. Hvis disse flader udvikler hver deres sandhed om session og completion, bliver fleksibel progression usikker.

\subsection{Lagdeling mellem brugerflade, state og data}

SwiftUI understøtter deklarative, state-drevne brugerflader \cite{apple_swiftui_docs}. I Træningsmester betyder det, at skærme viser aktuel session, aktivt program, træningsstatus og næste handling, men ikke selv er datamæssig autoritet. AppState, domænemodeller og repositories placerer logikken mellem brugerflade og backend.

Lagdelingen er nødvendig, fordi træningsflowet ikke bør afhænge af, at én skærm husker al kontekst. Når workoutstatus, trackerlog, completion og progression ligger i fælles state- og datalag, kan iOS, watchOS og Live Activity arbejde med samme træningsforløb. Tabel~\ref{tab:tm-container-architecture} viser komponentlagene i kondenseret form.

Lagdelingen gør også test og ændring lettere. Domænelogik for cycle runtime og completion kan vurderes adskilt fra den konkrete SwiftUI-visning. Det er vigtigt, fordi problemformuleringen handler om systemadfærd, ikke kun om om en bestemt knap eller skærm findes.

\subsection{Backend som sikkerhedsgrænse}

Backendens rolle er at være autoritativ data- og sikkerhedsgrænse. Klienten kan vise og starte handlinger, men den må ikke alene afgøre, hvilke data brugeren kan læse eller skrive. OWASP MASVS peger på samme princip for mobile apps, og Supabase RLS-dokumentationen beskriver, hvordan adgang kan begrænses efter bruger, rolle og kontekst \cite{owasp2024_masvs,supabase_rls_docs,supabase_securing_api_docs}.

Figur~\ref{fig:tm-security-boundary-summary} viser grænsen mellem mobil klient, RLS og Edge Functions. RLS beskytter almindelige brugerbundne læse- og skriveoperationer. Edge Functions afgrænser privilegerede handlinger, integrationer og nøgler. Det reducerer risikoen for, at en brugerfladefejl bliver til bredere dataadgang, men det dokumenterer ikke fuld sikkerhed i sig selv.

Denne grænse er særlig relevant, fordi træningsdata både er personlige og operationelle. Brugeren skal kunne logge hurtigt, men hurtig logging må ikke betyde, at klienten kan skrive eller læse på tværs af brugergrænser. Arkitekturen placerer derfor hastighed i brugerfladen og autoritet i backendlaget.

\subsection{Støtteflader under træning}

watchOS og Live Activity er arkitektoniske støtteflader. Apple beskriver Live Activities som løbende status for tidsfølsomme aktiviteter og Watch Connectivity som koordinering mellem iPhone og Apple Watch \cite{apple_activitykit_live_data_docs,apple_hig_live_activities,apple_watchconnectivity_docs}. I casen skal de reducere friktion under træning, ikke skabe parallelle datamodeller.

Figur~\ref{fig:tm-watch-sync-sequence} og Figur~\ref{fig:tm-live-activity-idempotency-sequence} viser, at støttefladerne afhænger af fælles session, auth-sync, snapshot og idempotente writes. Arkitekturens styrke er dermed ikke, at appen findes på flere flader, men at flere flader kan pege på samme træningsmodel.

Det giver også en klar begrænsning. Hvis watchOS eller Live Activity kun kan bygges, men ikke runtime-verificeres, er arkitekturargumentet stærkere end produktionsbeviset. Støttefladerne skal derfor forstås som en dokumenteret designretning med behov for yderligere test.
