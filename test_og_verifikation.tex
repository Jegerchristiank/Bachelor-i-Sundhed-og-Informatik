\section{Test og verifikation}

Test og verifikation vurderer, om de centrale tekniske påstande om TræningsMester er dokumenteret. Formålet er teknisk plausibilitet, ikke sundhedseffekt, brugeraccept eller produktionsklarhed.

Verifikationen læses derfor som et afgrænset teknisk evidensspor. Den kan vise, at centrale dele kan bygges, at udvalgt domænelogik er testet, og at brugerflader og arkitektur er dokumenteret. Den kan ikke vise, at brugerne bliver mere motiverede eller fastholdes bedre.

\subsection{Build, unit tests og centrale flows}

Tabel~\ref{tab:tm-test-traceability} viser, at iOS-klient, watchOS-target og Live Activity-target kan bygges, og at central forretningslogik er unit-testet. Det understøtter, at den udviklede app ikke kun er en visuel prototype, men har implementeret domænelogik for tracker, completion, cycle runtime og Live Activity-projection.

Buildstatus er dog afgrænset. Den dokumenterer, at targets kan kompileres i den gennemgåede tilstand, men ikke en uafhængig CI-proces, komplet installationsflow eller stabil runtime på tværs af enheder. Unit tests dokumenterer afgrænsede logikker; de viser ikke, at brugeren forstår flowet, eller at alle backend- og platformscenarier fungerer.

Unit tests passer især til de dele, hvor forventet adfærd kan beskrives præcist: payloads, trackerstatus, completion, cycle runtime og Live Activity-projection. De er svagere til spørgsmål om forståelighed, tempo og oplevet friktion, som kræver brugerafprøvning.

\subsection{Integrationstest og sikkerhedsbegrænsninger}

Integrationstests er mere begrænsede. Tabel~\ref{tab:tm-test-traceability} angiver, at Supabase anon read path kun er delvist bestået, fordi admin- og ikke-adminscenarier ikke er fuldt dækket. Derfor kan der ikke påstås fuld backend- eller rolleverifikation.

Det er især vigtigt for sikkerhedsmodellen. Figur~\ref{fig:tm-security-boundary-summary} og Tabel~\ref{tab:tm-rls-matrix} viser en arkitektur, hvor klient, RLS og Edge Functions har forskellige roller. OWASP MASVS og Supabase-dokumentationen understøtter princippet, men en model er ikke et sikkerhedsbevis \cite{owasp2024_masvs,supabase_rls_docs,supabase_securing_api_docs}. Admin-, træner-, tværbruger- og Edge Function-scenarier kræver mere målrettet test.

\subsection{Visuel dokumentation og åbne mangler}

Skærmbilleder og diagrammer dokumenterer, at centrale brugerflader og arkitekturspor findes. Figur~\ref{fig:tm-ios-home-ready} viser Home som næste-handling-flade, Figur~\ref{fig:tm-ios-tracker-active} viser aktiv tracker, og Figur~\ref{fig:tm-system-context} viser systemkonteksten. Det styrker forbindelsen mellem krav, implementering og synlige artefakter.

Rest-risici handler især om reproducerbarhed, backendintegration, rollebaseret adgang og runtime for watchOS og Live Activity. Åbne mangler svækker ikke arbejdet med TræningsMester, hvis de behandles åbent. De afgrænser blot, hvad der er dokumenteret: teknisk plausibilitet for centrale flows, ikke fuld sikkerhedscertificering, penetrationstest eller driftssikker produktionsmodenhed.

Den vigtigste konklusion fra verifikationen er derfor dobbelt. Kerneflowet har et dokumenteret teknisk grundlag, men flere integrations- og sikkerhedsscenarier skal testes, før appen kan vurderes som moden til bred drift.

\subsection{Verifikationsvurdering}

Verifikationen styrker bachelorens argument, fordi den forbinder tekstens påstande med konkrete artefakter: build, unit tests, skærmbilleder, diagrammer og sporbarhedstabeller. Det gør rapporten mere end en idébeskrivelse. Den viser, at centrale dele af designet er omsat til en fungerende teknisk struktur.

Samtidig er verifikationen ikke stærk nok til at bære de mest vidtgående påstande. De stærkeste konklusioner kan handle om implementeringsplausibilitet, arkitekturvalg og kendte begrænsninger. De svageste konklusioner ville være påstande om langtidseffekt, fuld sikkerhed eller stabil runtime på alle enheder. Derfor bruges verifikationen som et kontrolleret evidensspor og ikke som et kvalitetsstempel for færdig produktionsmodenhed.
